# PR-06 — AI Marketing Suggestion v1

## 1. PR Goal

เพิ่ม AI suggestion layer ให้ FlowBiz สามารถช่วยคิดโปร แนะนำข้อความ broadcast สรุปงานการตลาดรายวัน และสร้าง suggestion ที่นำไปใช้ต่อใน campaign/template ได้ โดยทุก suggestion ต้องตรวจสอบย้อนหลังและต้องมีการ approve ก่อนใช้ใน action สำคัญ

## 2. Why

นี่คือจุดต่างหลักของ FlowBiz

ถ้ามีแค่ CRM + automation:

* ระบบช่วย “ทำ”
  แต่ยังไม่ช่วย “คิด”

PR นี้ทำให้ product ขยับไปเป็น

> **AI Marketing Manager สำหรับคลินิก**

โดยไม่เสี่ยงเกินไป เพราะยังใช้แนวทาง **human-in-the-loop**

## 3. In Scope

### Data Model

* `content_suggestions`

### AI Suggestion Capability v1

รองรับ suggestion 4 กลุ่มหลัก

1. **Monthly Promotion Suggestion**
   ช่วยคิดโปรรายเดือน / แคมเปญตามช่วงเวลา

2. **Broadcast Copy Suggestion**
   ช่วยร่างข้อความ broadcast / follow-up / reactivation

3. **Daily Marketing Summary**
   สรุปว่าทีมควรทำอะไรวันนี้ เช่น follow lead ไหน, ดัน campaign ไหน

4. **Follow-up Copy Suggestion**
   ช่วยร่างข้อความตามบริบท lead/customer แบบมี context จาก CRM

### AI Workflow

* gather context จาก clinic / leads / customers / campaigns / due tasks
* build prompt ตาม suggestion type
* call LLM provider
* persist prompt version / context / output
* expose suggestion ให้ staff review
* approve / reject / regenerate suggestion
* optional convert suggestion → template / draft

### Approval Rule

* suggestion ทุกตัวที่ส่งผลต่อข้อความจริงหรือแคมเปญจริง ต้องมี manual approval ก่อนนำไปใช้
* ห้าม auto-send ใน PR นี้

## 4. Explicit Non-Goals

* auto campaign launch
* auto broadcast send
* AI วินิจฉัยด้านการแพทย์
* AI ตัดสินใจ budget allocation
* image/video generation pipeline
* editorial workflow เต็มรูปแบบ
* personalization engine ขั้นสูงต่อรายคนแบบ real-time
* training custom model

### Frontend

### Integration

## 7. Functional Acceptance Criteria

### Suggestion Generation

* staff สั่ง generate suggestion ได้ตาม type ที่รองรับ
* ระบบสร้าง suggestion โดยใช้ clinic context ที่เกี่ยวข้องได้
* output ถูกบันทึกพร้อม prompt version และ context snapshot

### Suggestion Review

* staff เปิดดู suggestion ได้
* approve ได้
* reject ได้
* regenerate ได้

### Suggestion Reuse

* suggestion ที่ approved สามารถแปลงเป็น template draft หรือ campaign draft ได้อย่างน้อย 1–2 path
* broadcast copy suggestion เอาไปใช้ใน messaging template ได้
* promotion suggestion เอาไปใช้สร้าง promotion draft ได้

### Daily Marketing Summary

* ระบบ generate สรุปรายวันได้
* summary ต้องมีลักษณะ actionable เช่น

  * leads ที่ควร follow
  * customers ที่ cycle due
  * campaign ที่ควร push
  * task ที่ overdue

### Safety

* ไม่มี path ที่ AI trigger send จริงเอง
* AI output ทุกชิ้น trace กลับได้ว่าใช้ prompt version ไหน
* มี budget/use guardrail พื้นฐานต่อ clinic หรือ system-wide

### Tenant Safety

* suggestion ผูก clinic ชัดเจน
* context ที่ใช้ต้องมาจาก clinic เดียวกัน
* ผู้ใช้ข้าม clinic ดู suggestion กันไม่ได้

## 8. Product Constraints

* AI output ต้องสั้น ใช้ได้จริง และเน้นเชิงปฏิบัติ
* หลีกเลี่ยง content ยาวแบบบทความใหญ่
* default tone ต้องเหมาะกับธุรกิจคลินิก ไม่ aggressive เกิน
* suggestion UI ต้องเน้น “นำไปใช้ต่อ” ไม่ใช่แค่แสดงข้อความเฉย ๆ

## 9. Risks

* AI hallucination
* suggestion generic เกิน ไม่ usable
* cost LLM สูง
* staff approve แล้วแต่ไม่ได้นำไปใช้จริง
* context quality ไม่ดี ทำให้ output ไม่น่าใช้

## 10. Risk Mitigation

* ใช้ prompt templates แบบ opinionated
* จำกัด suggestion types ให้แคบ
* context builder ต้องคัดเฉพาะข้อมูลที่มีประโยชน์
* manual approval mandatory
* token / usage cap ต่อ clinic
* store approved conversion rate เพื่อดูว่า suggestion แบบไหนใช้จริง

## 11. Rollback Plan

ถ้ามีปัญหา:

* ปิดบาง suggestion type
* ลดเหลือเฉพาะ daily summary กับ broadcast copy
* จำกัด generation เฉพาะ manager/owner
* fallback เป็น manual template recommendation แบบ rule-based

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* generate suggestion ตาม type หลักได้
* review/approve/reject/regenerate ได้
* เก็บ prompt version/context/output ได้
* suggestion ที่ approved ถูกนำไปใช้ต่อใน template/promotion draft ได้
* daily marketing summary ใช้งานได้จริง

# PR-08 — Analytics & Audit v1

## 1. PR Goal

สร้าง analytics และ audit foundation ให้ FlowBiz สามารถวัดผลลัพธ์ของ lead, messaging, automation, repeat revenue activity และตรวจสอบย้อนหลังการกระทำสำคัญของผู้ใช้และระบบได้อย่างน่าเชื่อถือ

## 2. Why

B2B SaaS ขายยากถ้าวัดผลไม่ได้
และระบบ automation ที่ไม่มี traceability จะเสียความเชื่อถือทันที

PR นี้จึงมี 2 หน้าที่:

### A. Analytics

ตอบว่า FlowBiz ช่วยอะไรธุรกิจบ้าง

### B. Audit

ตอบว่าใครทำอะไร เมื่อไร กับข้อมูลอะไร

สองอย่างนี้จำเป็นต่อการขาย และจำเป็นต่อการขยายไปสู่ลูกค้าระดับจริง

## 7. Functional Acceptance Criteria

### Event Tracking

* ระบบสร้าง activity event ได้เมื่อเกิด action หลัก
* event มี clinic/entity/event_type/occurred_at ครบ
* event feed สามารถใช้สร้าง summary metrics ได้

### Analytics Summary

* owner/manager เห็น KPI พื้นฐานได้อย่างน้อย:

  * lead ใหม่
  * leads by stage
  * follow-up due
  * messages sent
  * automation executions
  * cycle due customers

### Actionable Views

* มี list ของ leads overdue follow-up
* มี list ของ customers ที่ cycle due / inactive
* มี flow execution failures ให้ตรวจได้
* มี messaging failures ให้ตรวจได้

### Audit Logging

* action สำคัญถูกบันทึกใน audit log
* ระบุ actor ได้
* ระบุ before/after ได้ใน action ที่เกี่ยวกับ update
* filter audit log ตาม entity / action / actor / date ได้ขั้นพื้นฐาน

### Traceability

* เมื่อมีการ approve AI suggestion ต้อง trace ได้
* เมื่อมีการ enable/disable flow ต้อง trace ได้
* เมื่อมี message send ต้อง trace กลับไปยัง actor หรือ automation execution ได้

### Tenant Safety

* analytics/query ทุกตัว scoped ตาม clinic
* audit log ข้าม clinic มองไม่เห็นกัน
* dashboard ไม่ aggregate ข้าม clinic ใน user-facing path

## 8. Product Constraints

* dashboard ต้องเน้น “operational clarity” มากกว่า “สวย”
* metric definitions ต้อง canonical และไม่เปลี่ยนไปมา
* หลีกเลี่ยง dashboard ที่มีตัวเลขเยอะแต่ไม่มี action
* audit UI ต้องอ่านง่าย ไม่ technical เกินจำเป็น

## 9. Risks

* metric นิยามไม่ตรงกัน
* event tracking ไม่ครบ ทำให้ตัวเลขไม่น่าเชื่อถือ
* audit log โตเร็ว
* query dashboard ช้า
* activity feed กลายเป็น noisy log

## 10. Risk Mitigation

* define canonical metrics ชัดก่อน build
* event naming convention เดียวทั้งระบบ
* เก็บเฉพาะ action สำคัญจริง
* แยก summary query กับ raw feed query
* ใส่ indexes ตาม clinic/date/entity/action
* ใช้ aggregation jobs เมื่อจำเป็น แต่อย่าเริ่มใหญ่เกิน

## 11. Rollback Plan

ถ้ามีปัญหา:

* ลด dashboard เหลือ summary cards + 2 actionable lists
* ลด audit scope เหลือ critical actions ก่อน
* เก็บ raw events ต่อ แต่ปิดบาง widgets ที่ยังไม่นิ่ง
* move expensive metrics เป็น daily aggregation ภายหลัง

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* event tracking พื้นฐานทำงานได้
* dashboard summary/actionable views ใช้งานได้จริง
* audit logs สำหรับ action สำคัญใช้งานได้
* owner/manager เริ่มวัดผลและตรวจย้อนหลังได้
* system พร้อมสำหรับการขายแบบ evidence-based มากขึ้น
