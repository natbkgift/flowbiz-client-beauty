# FlowBiz MVP — Final PR Summary (PR-01 to PR-03)

## หลักการรวมก่อนเริ่ม Build

### Product Slice ของรอบนี้

รอบนี้ยัง **ไม่พยายามทำ product ทั้งหมด**
เป้าหมายคือวางฐานให้ระบบเริ่มทำงานได้จริงใน 3 เรื่อง

1. **รองรับหลายคลินิกอย่างปลอดภัย**
2. **มี Lead CRM แกนหลัก**
3. **มีระบบส่งข้อความและบันทึก log ได้**

### Non-Goals ของรอบนี้

* automation engine
* AI suggestion

* analytics dashboard เชิงลึก
* customer lifecycle เต็มระบบ
* custom workflow builder
* medical records / billing / booking engine เต็มรูปแบบ

### Build Principle

* 1 PR = 1 เป้าหมาย
* schema ต้อง minimal แต่ไม่ตันในอนาคต
* ทุกอย่างต้อง tenant-safe
* ทุก write path สำคัญควร audit ได้ในระดับพื้นฐาน
* ห้าม refactor นอก scope

# PR-01 — Project Foundation & Multi-Tenant Base

## 1. PR Goal

วางโครงสร้างระบบพื้นฐานของ SaaS แบบหลายคลินิก โดยให้ auth, tenant isolation, role access, และ data ownership ชัดตั้งแต่ต้น

## 2. Why

ถ้า foundation นี้ไม่แน่น จะเกิดปัญหาใหญ่ทันทีเมื่อมี clinic มากกว่า 1 แห่ง เช่น

* ข้อมูลข้าม tenant
* สิทธิ์ผู้ใช้ไม่ชัด
* schema โตแล้วแก้ยาก
* query และ service layer ไม่มีมาตรฐานร่วม

PR นี้จึงเป็นฐานของทุก PR ถัดไป

## 3. In Scope

### Data Model

* `clinics`
* `users`
* `clinic_users`

### System Capability

* login / auth baseline
* current clinic context
* role model baseline
* tenant-aware request handling
* tenant-aware data access pattern
* base migration / seed structure

### Access Model

รองรับ role อย่างน้อย:

* owner
* manager
* sales
* marketing
* staff

### Platform Rules

* ทุก business table หลังจากนี้ต้องมี `clinic_id` หรือ tenant ownership path ที่ชัด
* ทุก query ผ่าน service/repository pattern ที่ enforce tenant scope
* ทุก endpoint หลักต้อง resolve clinic context ได้

## 4. Explicit Non-Goals

* SSO
* permission matrix แบบละเอียดมาก
* internal super admin UI เต็มรูปแบบ
* invite flow ซับซ้อน
* audit log เต็มระบบ
* multi-branch hierarchy
* subscription billing

## 5. Files / Modules Expected to Change

### Backend

* auth module
* clinic module
* user module
* clinic membership / role module
* request context / middleware
* db migration setup
* base repository/service utilities

### Frontend / Admin

* login page / session bootstrap
* clinic switcher ถ้าจำเป็น
* current user / current clinic state

### Infra / Config

* env config for auth/session/db
* migration command / seed command

## 6. Data Contracts

### `clinics`

ขั้นต่ำควรมี

* id
* name
* slug
* plan
* status
* timezone
* created_at
* updated_at

### `users`

ขั้นต่ำควรมี

* id
* email
* name
* password_hash
* status
* last_login_at
* created_at
* updated_at

### `clinic_users`

ขั้นต่ำควรมี

* id
* clinic_id
* user_id
* role
* status
* created_at
* updated_at

## 7. Functional Acceptance Criteria

### Auth / Session

* ผู้ใช้ login ได้
* session ระบุ current user ได้
* resolve current clinic ได้อย่างปลอดภัย

### Tenant Isolation

* ผู้ใช้เห็นข้อมูลเฉพาะ clinic ที่ตัวเองเป็นสมาชิก
* query ที่ไม่มี clinic scope ไม่ผ่าน path ปกติ
* ถ้ามีการเรียก resource ต่าง clinic ต้องถูกปฏิเสธ

### Membership / Roles

* เพิ่ม user เข้า clinic ได้
* role ถูกเก็บและถูกอ่านได้
* API สามารถเช็ค role ขั้นพื้นฐานได้

### Developer Experience

* migration รันได้
* seed clinic/user ตัวอย่างได้
* local dev setup ใช้งานได้

## 8. Technical Constraints

* ใช้ modular monolith
* ใช้ PostgreSQL เป็น source of truth
* หลีกเลี่ยง abstraction ที่มากเกินในรอบแรก
* ห้ามทำ generic permission engine ใหญ่เกินจำเป็น

## 9. Risks

* tenant leakage จาก query ที่ลืมใส่ scope
* role model ใหญ่เกินจน build ช้า
* auth flow พาให้ frontend/blocking เยอะ

## 10. Risk Mitigation

* บังคับ service/repository รับ `clinic_id` ทุกครั้ง
* ใช้ middleware/context เดียวกันทั้งระบบ
* เริ่มจาก role-based access แบบหยาบก่อน
* เพิ่ม test สำหรับ cross-tenant access

## 11. Rollback Plan

ถ้ามีปัญหา

* ปิด multi-clinic switching ชั่วคราว
* ใช้ single-clinic-per-user path ชั่วคราวได้
* ลด role เหลือ owner / staff ก่อน

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* login ได้
* current clinic context ทำงานได้
* membership/role baseline ใช้งานได้
* tenant-safe path พร้อมสำหรับ PR-02

# PR-02 — Lead CRM Core

## 1. PR Goal

สร้าง Lead CRM แกนหลักให้คลินิกสามารถเก็บ ติดตาม จัด stage และใส่บริบทของ lead ได้ในระบบเดียว

## 2. Why

Lead CRM คือหัวใจของ product
ถ้ายังไม่มี lead model ที่ใช้งานจริง automation, messaging, campaign และ analytics จะไม่มีฐานที่เชื่อถือได้

PR นี้ต้องทำให้ทีมคลินิกเริ่มใช้ระบบเป็น “ที่เดียวสำหรับดู lead” ได้

## 3. In Scope

* `lead_interests`
* `notes`
* `tags`
* `entity_tags`

### CRM Capability

* create lead
* edit lead
* list/search/filter leads
* view lead detail
* change lead stage/status
* add interest
* add tag
* add note
* assign owner

### Timeline / Activity (ขั้นต่ำ)

* activity event พื้นฐานสำหรับ lead change สำคัญ หรือใช้ note/event log แบบง่ายก่อน

### Minimal UI / API

## 4. Explicit Non-Goals

## 7. Functional Acceptance Criteria

### Lead CRUD

* สร้าง lead ใหม่ได้
* แก้ไขข้อมูล lead ได้
* เปิดหน้า detail แล้วเห็นข้อมูลหลักครบ

### Lead Tracking

* เปลี่ยน status/stage ได้
* assign owner ได้
* บันทึก next follow-up ได้
* filter ตาม stage/status/owner/source ได้

### Lead Context

* เพิ่ม interest ได้
* เพิ่ม tag ได้
* เพิ่ม note ได้
* แสดง note/tag/interest ในหน้า detail ได้

### Tenant Safety

* lead ทุกตัวผูก clinic ชัด
* user ต่าง clinic เข้าถึงไม่ได้

### Data Integrity

* owner ต้องเป็น member ของ clinic เดียวกัน
* note/tag ที่ผูก entity ต้องตรง clinic เดียวกัน
* stage/status ใช้ enum เดียวกันทั้งระบบ

## 8. UX / Product Constraints

* ฟอร์มต้องไม่ยาวเกินไป
* field ที่ไม่ critical ให้ optional
* stage transition ไม่ต้อง enforce แบบ workflow engine
* list page ต้องใช้งานจริงกับทีม sale ได้

## 9. Risks

* lead model ใหญ่เกินจนทีมกรอกไม่ไหว
* stage เยอะเกินจนใช้งานจริงสับสน
* duplicate lead เยอะ แต่ยังไม่มี dedupe

## 10. Risk Mitigation

* เริ่มจาก minimal required fields
* ใช้ stage ชุดเดียวทั้งระบบ
* duplicate handling ใช้ manual ก่อน
* เก็บ `source_ref`, `line_user_id`, `phone`, `email` เพื่อรองรับ dedupe รอบถัดไป

## 11. Rollback Plan

ถ้าการใช้งานจริงซับซ้อนเกิน:

* ลด fields ใน form
* ซ่อน optional sections
* ลดจำนวน stage ที่แสดงใน UI โดยยังเก็บ schema เดิมไว้ได้

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* ทีมสามารถสร้างและดู lead ได้
* มี stage/status/tag/note/interest ครบ
* หน้า lead list/detail ใช้งานได้จริง
* ระบบพร้อมสำหรับ PR-03 ที่จะผูก messaging

# PR-03 — Messaging & Template Foundation

## 1. PR Goal

วางรากฐานระบบข้อความของ FlowBiz ให้สามารถเก็บ template, map contact identity, ส่งข้อความออกผ่าน channel หลัก, และมี message log ที่ตรวจสอบย้อนหลังได้

## 2. Why

Messaging คือจุดเชื่อมสำคัญของ product
ถ้าไม่มีระบบนี้ ต่อให้มี CRM ก็ยังไม่เกิดคุณค่าเชิง automation หรือ follow-up จริง

PR นี้ยังไม่เน้น “broadcast ใหญ่”
แต่ต้องทำให้ระบบสามารถ **ส่งข้อความแบบควบคุมได้และตรวจสอบได้**

## 3. In Scope

### Data Model

* `channels`
* `contact_identities`
* `message_templates`
* `outbound_messages`

### Messaging Capability

* register/manage 1 primary messaging channel
* map lead/customer กับ external recipient identity
* create/edit template
* render template แบบพื้นฐาน
* manual send message จาก lead/customer context
* save outbound message log
* track delivery/send state ขั้นพื้นฐาน

### Product Path

* เลือก recipient จาก lead หรือ customer entity
* ส่งข้อความออกผ่าน 1 channel หลัก
* กลับมาดูประวัติ outbound ได้

## 4. Explicit Non-Goals

* inbox/conversation center เต็มรูปแบบ
* inbound webhook orchestration เต็มระบบ
* multi-channel switching แบบสมบูรณ์
* campaign broadcast batching
* retries/queue sophistication สูง
* A/B template testing
* AI generated copy

## 5. Files / Modules Expected to Change

### Backend

* messaging/channels module
* contact identity module
* template module
* outbound message service
* provider adapter สำหรับ 1 channel หลัก
* delivery status mapper
* lead/customer integration hooks

### Frontend

* template list/create/edit UI
* manual send action ใน lead/customer detail
* outbound message history section

### Infra / Integration

* provider credentials config
* secure secrets handling
* minimal webhook/config verification ถ้าจำเป็นต่อ provider

## 7. Functional Acceptance Criteria

### Channel Foundation

* clinic มี channel หลักได้อย่างน้อย 1 channel
* config ของ channel ถูกเก็บแยกต่อ clinic
* channel status ใช้เปิด/ปิดได้

### Contact Identity

* ผูก lead กับ external recipient ได้
* มี primary identity ต่อ channel ได้
* ใช้ identity นี้ตอนส่งข้อความได้

### Templates

* สร้าง template ได้
* แก้ template ได้
* ระบุ category และ variables ได้
* render content ขั้นพื้นฐานได้

### Manual Send

* staff ส่งข้อความจาก lead/customer detail ได้
* ระบบเลือก recipient identity ได้ถูกต้อง
* ส่งสำเร็จแล้วสร้าง outbound log ได้
* ถ้าส่งไม่สำเร็จ มี failed status และเหตุผลเบื้องต้น

### Traceability

* เปิดดู outbound message history ได้
* รู้ว่าใครส่ง ส่งให้ใคร ผ่าน channel อะไร เมื่อไร
* ถ้าใช้ template ต้องรู้ว่าใช้ template ไหน

### Tenant Safety

* template/channel/identity/message ต้องอยู่ clinic เดียวกันทั้งหมด
* ห้ามใช้ template ข้าม clinic
* ห้ามส่งออกผ่าน channel คนละ clinic

## 8. Technical Constraints

* เริ่มจาก 1 primary provider/channel ก่อน
* manual send path ต้องเรียบง่าย
* provider adapter ต้องแยกชั้นจาก domain logic
* อย่ารีบทำ orchestration หรือ broadcast scheduler ใน PR นี้

## 9. Risks

* mapping recipient ผิดคน
* state ของ outbound message ไม่ครบ
* provider error handling ไม่สม่ำเสมอ
* template variables พังตอน render

## 10. Risk Mitigation

* validate identity ก่อนส่งเสมอ
* เก็บ rendered content ลง log ทุกครั้ง
* status machine ให้เล็กและชัด
* template variable rendering ใช้ strict validation

## 11. Rollback Plan

ถ้าปัญหาที่ integration หนักเกิน:

* ใช้ simulated provider ใน dev/staging
* ปิด manual send สำหรับบาง role
* จำกัดให้ส่งจาก lead detail ก่อน customer detail

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* clinic ตั้งค่า channel หลักได้
* สร้าง template ได้
* ผูก recipient identity ได้
* staff ส่งข้อความจาก lead context ได้จริง
* outbound log ตรวจสอบย้อนหลังได้
