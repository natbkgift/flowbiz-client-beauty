# FlowBiz MVP — Final PR Summary (PR-04, PR-05, PR-07)

## หลักการรวมก่อนเริ่ม Build

### เป้าหมายของรอบนี้

หลังจากมี

* multi-tenant base
* lead CRM
* messaging foundation

รอบนี้ต้องทำให้ระบบเริ่ม **ลงมือแทนทีมคลินิกได้จริง** ใน 3 ระดับ

1. **มี automation engine ที่เชื่อถือได้**
2. **มี flow สำเร็จรูปที่ช่วยงานและสร้างรายได้**
3. **เริ่มรองรับ repeat revenue จากลูกค้าเก่า**

### Product Boundary ของรอบนี้

รอบนี้ยังไม่ทำ “platform automation แบบใหญ่”
แต่ทำให้พร้อมใช้งานจริงสำหรับ use case หลักของคลินิก

### Non-Goals รวมของรอบนี้

ยังไม่ทำใน PR-04 / PR-05 / PR-07

* visual drag-and-drop flow builder
* omnichannel orchestration เต็มรูปแบบ
* AI ตัดสินใจส่งข้อความเองอัตโนมัติ
* predictive scoring ขั้นสูง
* medical workflow
* billing / package accounting
* advanced analytics
* custom automation marketplace

### Build Principle

* engine ต้อง **idempotent**
* ทุก execution ต้อง **trace ได้**
* flow สำเร็จรูปต้อง **เปิด/ปิดเป็นรายคลินิกได้**
* customer/treatment layer ต้องเป็น **marketing-oriented data model** ไม่หลุดไปเป็น EMR

# PR-04 — Automation Engine v1

## 1. PR Goal

สร้าง automation engine รุ่นแรกของ FlowBiz ให้รองรับ trigger-based และ schedule-based automation แบบพื้นฐาน โดยสามารถรัน step ตามลำดับ เก็บ execution log และสร้าง task/reminder/message action ได้อย่างเชื่อถือได้

## 2. Why

นี่คือหัวใจของ product

ถ้าไม่มี automation engine:

* follow-up จะยังเป็น manual
* reminder จะยังไม่เป็นระบบ
* MVP automation pack จะไม่มีฐานรองรับ
* product จะยังไม่ต่างจาก CRM + messaging ธรรมดา

PR นี้จึงต้องทำให้ระบบมี “execution backbone” ที่มั่นคงก่อน

## 3. In Scope

### Data Model

* `automation_flows`
* `automation_steps`
* `automation_executions`
* `automation_tasks`
* `reminders`

### Engine Capability

* define flow metadata
* define ordered steps
* trigger execution จาก event ที่ระบบรู้จัก
* run sequential steps แบบพื้นฐาน
* support delayed/wait steps
* support step actions ชุดแรก
* persist execution state
* retry-safe execution pattern
* execution history / logs

### Supported Trigger Types v1

ขั้นต่ำควรรองรับ:

* entity event trigger
  เช่น `lead.created`, `lead.stage_changed`, `treatment.completed`
* scheduled due trigger
  เช่น reminder due, follow-up due

### Supported Step Types v1

ขั้นต่ำควรรองรับ:

* `wait`
* `send_message`
* `create_task`
* `add_tag`
* `remove_tag`
* `change_stage`
* `create_reminder`
* `notify_user`

### Runtime Capability

* worker/background job processing
* delayed scheduling
* execution resume after wait
* per-step status tracking
* basic failure handling

## 4. Explicit Non-Goals

* drag-and-drop builder
* nested branching ซับซ้อน
* user-defined scripting
* conditional expression engine ขนาดใหญ่
* complex SLA engine
* flow template marketplace
* auto-optimization by AI
* inbound conversation branching ขั้นสูง

## 5. Files / Modules Expected to Change

### Backend Core

* automation module
* flow repository/service

* scheduler / queue worker
* event dispatcher / internal event hooks
* reminder/task service integration
* messaging integration adapter

### Data / Infra

* idempotency guard helpers

### Frontend / Admin

## 7. Functional Acceptance Criteria

### Flow Lifecycle

* สร้าง/seed flow สำเร็จรูปได้
* เปิด/ปิด flow ได้
* flow ที่ disabled ต้องไม่รัน execution ใหม่

### Triggering

* เมื่อ event ที่รองรับเกิดขึ้น ระบบสร้าง execution ได้
* execution ต้องผูก clinic/entity/flow ชัดเจน
* trigger ซ้ำต้องไม่ทำให้ execution ซ้ำแบบไม่ตั้งใจ

### Step Execution

* step order ทำงานตามลำดับ
* `wait` ทำให้ flow หยุดและกลับมาทำต่อได้
* `send_message` เรียก messaging foundation ได้
* `create_task` และ `create_reminder` ทำงานได้จริง
* `change_stage` และ `add/remove_tag` update CRM ได้

### Traceability

* execution status ดูได้
* step ที่สำเร็จ/ล้มเหลวระบุได้
* failure reason เบื้องต้นถูกบันทึก
* เห็นว่า flow ไหน trigger จาก event อะไร

### Reliability

* worker restart แล้ว execution ที่ค้างยัง recover ได้
* step เดิมไม่ถูกยิงซ้ำโดยไม่มี idempotency guard
* failure ของ step หนึ่งไม่ทำให้ข้อมูลพังทั้ง execution

### Tenant Safety

* event, flow, execution, entity ต้องอยู่ clinic เดียวกัน
* ห้าม execution ข้าม clinic

## 8. Technical Constraints

* ใช้ modular monolith + async workers
* ใช้ PostgreSQL + Redis
* ใช้ internal event dispatch แบบเรียบง่าย
* ไม่สร้าง generic workflow DSL ใหญ่เกินจำเป็น
* step types ต้องเป็น fixed enum ก่อน

## 9. Risks

* duplicate execution
* step ยิงซ้ำ
* race condition ระหว่าง worker
* state machine execution ไม่ชัด
* scheduled jobs สะสมจน debugging ยาก

## 10. Risk Mitigation

* idempotency key ต่อ execution/step
* execution lock ระดับ worker
* status model ให้เล็กและ deterministic
* log transition สำคัญทุกครั้ง
* delayed step ใช้ scheduler pattern เดียวทั้งระบบ

## 11. Rollback Plan

ถ้ามีปัญหา:

* ปิด flow activation เป็นราย clinic
* disable step types ที่เสี่ยง เช่น change_stage
* เหลือเฉพาะ create_task / reminder ก่อน
* fallback เป็น manual reminder mode

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* event trigger execution ได้
* sequential step execution ทำงานได้
* delayed step resume ได้
* execution/task/reminder/message action trace ได้
* พร้อมรองรับ MVP flow pack ใน PR-05

# PR-05 — MVP Automation Pack

## 1. PR Goal

นำ automation engine มาแปลงเป็น **flow สำเร็จรูปที่ใช้งานได้จริง** สำหรับคลินิก โดยเน้น use case ที่ช่วยปิด lead, ลด no-show, เพิ่ม review และเพิ่ม repeat action

## 2. Why

ถ้ามี engine แต่ไม่มี flow สำเร็จรูป ลูกค้าจะยังไม่รู้สึกถึงคุณค่า
PR นี้คือการเปลี่ยนจาก “platform capability” เป็น “product outcome”

เป้าหมายคือให้คลินิกเปิดใช้งานแล้วรู้ทันทีว่า:

* วันนี้ต้อง follow ใคร
* lead ไหนกำลังตกหล่น
* คนที่จองแล้วต้องเตือนเมื่อไร
* ใครควรถูกขอรีวิว
* ใครควรถูกกระตุ้นให้กลับมา

## 3. In Scope

### MVP Flow Pack

ให้ลง flow สำเร็จรูป 8 ตัวตามที่ตกลงไว้

1. New Lead Welcome
2. Uncontacted Lead Alert
3. Lead Qualification Nurture
4. Consult Reminder Flow
5. No-Show Recovery
6. Review Request
7. Botox Cycle Reminder
8. Daily Marketing Reminder

### Product Capability

* seed/create predefined flows ต่อ clinic
* เปิด/ปิด flow ราย clinic ได้
* มี default template / default delays / default rules
* execution logs อ่านเข้าใจได้
* staff เห็น task/reminder ที่เกิดจาก flow
* flow summary view ขั้นพื้นฐาน

### Configurability v1

อนุญาตเฉพาะ config ที่จำเป็น เช่น

* on/off
* delay days/hours
* assigned role/user
* selected template
* quiet hours / send suppression เบื้องต้น
* review delay / cycle days บางกรณี

## 4. Explicit Non-Goals

* custom flow editor สำหรับลูกค้า
* branching strategy ขั้นสูง
* campaign optimizer
* auto A/B testing
* AI-generated flow logic
* multi-language flow orchestration เต็มรูปแบบ
* omnichannel sequence builder

## 5. Files / Modules Expected to Change

### Backend

* automation presets / seeds
* flow configuration service
* clinic flow activation service
* reminder / task aggregation service
* daily summary generator
* messaging templates seed pack
* treatment-cycle rule helper

### Frontend

* predefined flow list
* flow activation/config UI
* execution summary / recent activity
* staff reminder list
* daily marketing summary view

### CRM / Messaging Integration

* lead event hooks
* consult status hooks
* no-show hooks
* review request hooks
* customer treatment cycle hooks

## 6. Flow Contracts

## Flow 1 — New Lead Welcome

**Trigger:** lead.created
**Default Action:** ส่ง welcome message + create follow-up reminder

## Flow 2 — Uncontacted Lead Alert

**Trigger:** lead.created with no outbound within threshold
**Default Action:** create staff task + optional notify manager

## Flow 3 — Lead Qualification Nurture

**Trigger:** lead.stage = inquiry
**Default Action:** timed sequence of education/review/CTA

## Flow 4 — Consult Reminder Flow

**Trigger:** consult booked
**Default Action:** confirmation + reminder before appointment

## Flow 5 — No-Show Recovery

**Trigger:** appointment marked no_show
**Default Action:** rebooking message + staff task

## Flow 6 — Review Request

**Trigger:** treatment completed + configured waiting period
**Default Action:** request review + reminder if no response

## Flow 7 — Botox Cycle Reminder

**Trigger:** cycle due from treatment history / configured due date
**Default Action:** send reminder + optional sales task

## Flow 8 — Daily Marketing Reminder

**Trigger:** daily scheduled run
**Default Action:** summarize due follow-ups / inactive leads / campaign actions

## 7. Functional Acceptance Criteria

### Activation & Configuration

* แต่ละ clinic เปิด/ปิด flow ได้
* เปลี่ยนค่า config พื้นฐานได้โดยไม่ต้องแก้ code
* clinic ใหม่สามารถ seed flow pack ได้

### Execution

* flow ทั้ง 8 ตัว trigger ได้จาก event ที่เกี่ยวข้อง
* message, reminder, task ที่เกิดขึ้นตรงตาม preset
* default template ใช้งานได้
* execution log อ่านแล้วเข้าใจว่า flow ทำอะไรไปแล้วบ้าง

### Staff Experience

* staff เห็น task/reminder จาก flow ได้
* daily marketing summary แสดงรายการ action สำคัญได้
* manager มองเห็น lead ตกหล่น / no-show / due reminder ได้

### Safety & Control

* flow ที่ปิดอยู่ต้องไม่รัน
* มี suppression rule เบื้องต้นกันยิงข้อความเกินจำเป็น
* review request / marketing reminder ไม่ยิงซ้ำถี่เกินกำหนด

### Business Usefulness

* clinic สามารถเดโม 8 flow นี้ให้เห็น value ได้จริง
* owner เข้าใจได้ว่า flow ไหนช่วยรายได้ / งานขาย / การกลับมาซื้อซ้ำ

## 8. Product Constraints

* flow pack ต้อง opinionated
* config ต้องไม่เยอะจนใช้งานยาก
* template content ใช้ได้จริง ไม่ generic เกิน
* summary/dashboard ของ flow รอบนี้ให้เน้น “actionable” ไม่ใช่ analytics สวยงาม

## 9. Risks

* flow เยอะเกินจนทีมงง
* message fatigue
* default rule ไม่ตรงบริบทบางคลินิก
* consult / no-show event model ยังไม่นิ่ง

## 10. Risk Mitigation

* เริ่มจาก 8 flow เท่านั้น
* flow ทุกตัวมี toggle
* มี cooldown / quiet hours เบื้องต้น
* config จำกัดเฉพาะ field สำคัญ
* เอกสาร/tooltip อธิบาย flow ชัดเจน

## 11. Rollback Plan

ถ้ามีปัญหา:

* ปิดบาง flow เป็นราย clinic
* ลดเหลือ core pack 4 flow แรกชั่วคราว
* ปิด auto-send ของ flow ที่เสี่ยง และให้ create_task แทน

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* 8 flow สำเร็จรูปเปิดใช้ได้จริง
* trigger และ execution ทำงานได้
* staff เห็น task/reminder/summary ได้
* product เริ่ม demo เป็น “automation outcome” ได้ ไม่ใช่แค่ engine

# PR-07 — Customer & Repeat Revenue Layer

## 1. PR Goal

เพิ่ม customer profile และ treatment history เชิงการตลาด เพื่อให้ FlowBiz รองรับรายได้จากลูกค้าเก่า ผ่าน cycle reminders, reactivation และ repeat revenue workflows

## 2. Why

ถ้าระบบดูแลได้แค่ lead ใหม่ จะสร้างมูลค่าไม่เต็ม
คลินิกความงามและศัลยกรรมมีรายได้จำนวนมากจาก:

* botox เติมซ้ำ
* filler renewal
* skin program ต่อเนื่อง
* ลูกค้าเก่ากลับมาทำเพิ่ม
* VIP/reactivation

PR นี้คือฐานของ **LTV automation**

## 3. In Scope

### Data Model

* `customers`
* `customer_treatments`

### Capability

* convert lead → customer
* maintain customer profile
* add/edit treatment history เชิงการตลาด
* calculate/store `next_recommended_at`
* support treatment cycle reminder use case
* generate reactivation candidate views/lists
* support repeat-revenue related automation hooks

### Lifecycle Support

* ลูกค้าที่ปิดเคสแล้วไม่หายไปจากระบบ
* ทีมเห็น last visit / treatment / cycle due ได้
* flow อย่าง Botox Cycle Reminder และ customer reactivation มีฐานข้อมูลใช้งานจริง

## 4. Explicit Non-Goals

* full EMR / medical record
* doctor notes เชิงแพทย์
* prescription
* inventory consumption
* package accounting เต็มรูปแบบ
* surgery room workflow
* outcome measurement ทางการแพทย์

## 5. Files / Modules Expected to Change

### Backend

* customers module
* customer conversion service
* treatment history module
* cycle calculation helper
* repeat revenue rule helpers
* CRM integration for lead→customer conversion
* automation event hooks for customer/treatment events

### Frontend

* customer list/detail page
* convert lead to customer action
* treatment history section
* repeat due / reactivation candidate list

### Analytics / Summary Support

* customer status aggregation
* last visit / next recommended / inactivity flags

## 7. Functional Acceptance Criteria

### Lead → Customer Conversion

* แปลง lead เป็น customer ได้
* customer record ผูก lead เดิมได้
* history สำคัญของ lead ถูกเข้าถึงต่อได้
* conversion ไม่ทำให้ lead data สูญหาย

### Customer Profile

* ดู customer detail ได้
* เห็น last visit / lifetime value / status ได้
* ค้นหา/filter customer ได้ตาม status / visit recency

### Treatment History

* เพิ่ม treatment history ได้
* ระบุ treatment type / performed date / cycle days ได้
* ระบบคำนวณหรือเก็บ `next_recommended_at` ได้
* แก้ treatment history ได้แบบควบคุมได้

### Repeat Revenue Support

* ระบุลูกค้าที่ cycle due ได้
* ระบุลูกค้าที่ inactive ตาม threshold ได้
* flow ใน automation layer ใช้ customer/treatment event เหล่านี้ได้

### Tenant Safety

* customer และ treatment อยู่ clinic เดียวกัน
* lead→customer conversion ข้าม clinic ไม่ได้

### Data Integrity

* customer_treatments ต้องอ้างถึง customer ถูกต้อง
* `last_visit_at` อัปเดตตาม rule ที่กำหนด
* cycle defaults ใช้ได้อย่างสม่ำเสมอ

## 8. Product Constraints

* treatment model ต้องเป็นภาษาธุรกิจ/การตลาด ไม่ใช่ medical complexity
* อย่าบังคับให้ทีมกรอก field มากเกิน
* cycle handling ใช้ default simple rules ก่อน
* LTV ไม่ต้องสมบูรณ์ระดับบัญชี แค่พอใช้เชิงการตลาด

## 9. Risks

* treatment data ไม่ครบ
* cycle ของแต่ละ treatment ต่างกันและไม่แม่น
* ทีมใช้ customer/treatment model แบบไม่สม่ำเสมอ
* ระบบเริ่ม drift ไปเป็น clinic operations system

## 10. Risk Mitigation

* ใช้ treatment presets / category defaults
* allow manual override ของ cycle
* ฟอร์ม treatment ให้สั้น
* define ชัดว่าข้อมูลนี้ “for marketing automation” ไม่ใช่เวชระเบียน

## 11. Rollback Plan

ถ้ามีปัญหา:

* จำกัด treatment category ที่รองรับในรอบแรก
* ปิด auto-cycle calculation แล้วใช้ manual next recommended date
* จำกัด conversion flow ให้เฉพาะ owner/manager

## 12. Exit Criteria

PR นี้ถือว่าเสร็จเมื่อ:

* lead→customer conversion ทำงานได้
* customer profile ใช้งานได้จริง
* treatment history ใช้งานได้จริง
* cycle due และ reactivation candidates มองเห็นได้
* repeat-revenue automation มีฐานข้อมูลพร้อมใช้
